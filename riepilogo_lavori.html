2. Bug Fix Critici

2.1 Auth callback – sessione non persistita dopo conferma email

Il file src/app/auth/callback/route.ts scriveva i cookie di sessione sulla request invece che sulla response, causando la perdita della sessione subito dopo il login via link email.

// PRIMA (sbagliato)
request.cookies.set(name, value)

// DOPO (corretto)
response.cookies.set(name, value, options)  // ← options include SameSite, HttpOnly, ecc.

2.2 Matricola socio non collegata all'account

La matricola inserita in fase di registrazione non veniva mai trasmessa al callback di conferma email. Fix: aggiunta come parametro nella URL di redirect.

const callbackUrl = new URL(`${location.origin}/auth/callback`)
if (matricola) callbackUrl.searchParams.set('matricola', matricola)

2.3 Font CSS – riferimento circolare

In globals.css le variabili CSS dei font puntavano a se stesse (var(--font-sans) si auto-referenziava). Fix: agganciate alle variabili generate da Next.js.

/* PRIMA */
--font-sans: var(--font-sans);   /* circolare! */
--font-mono: var(--font-geist-mono);  /* inesistente */

/* DOPO */
--font-sans: var(--font-ibm-sans);
--font-mono: var(--font-ibm-mono);

2.4 Messaggio di errore login mostrato in verde

La logica per scegliere il colore del messaggio controllava message.includes('Controlla'), ma quella stringa appariva anche nei messaggi di errore. Fix: controllo invertito su keyword di successo.

2.5 Hydration mismatch – particelle animate

La homepage usava Math.random() per posizionare le particelle decorative, causando un mismatch tra server-side rendering e client-side. Fix: array deterministico pre-calcolato.

2.6 TypeScript error – Recharts Tooltip formatter

Il tipo Formatter<ValueType, NameType> di Recharts non accettava la type annotation esplicita. Fix: rimossa la annotazione lasciando inferire a TypeScript.

// PRIMA – errore di build
formatter={(value: number) => [value, 'Soci']}

// DOPO – corretto
formatter={(value) => [value, 'Soci']}

5. Autenticazione e Sicurezza

5.1 Middleware (middleware.ts)

Le restrizioni di accesso alle route sono state temporaneamente rimosse per permettere la navigazione senza account. Il middleware aggiorna solo la sessione:

// Solo refresh sessione, nessuna protezione route
await supabase.auth.getUser();
return response;

Le policy originali per authenticated bloccavano l'accesso ai visitatori non loggati (la web app è attualmente accessibile senza login per scelta di design). Aggiunte policy anon permissive su tutte le tabelle di lettura dati pubblici. Inserimenti/modifiche restano protetti.

La policy originale "Soci possono vedere i propri dati" (che mostrava solo il proprio record) è stata sostituita con una policy permissiva, necessaria per il funzionamento della dashboard amministrativa.

8-bis. Fix Critico: Attività ↔ Club Critico

Bug nei dati esportati da Lions: tutte le attività venivano associate al club ABETONE MONTAGNA PISTOIESE invece che ai loro club reali.

8-bis.1 Diagnosi

Il file report.csv di Lions contiene 80 nomi club distinti nella colonna Sponsor: Nome account, ma 1 solo ID (001Ns00000EDgRMIA1 = ABETONE) ripetuto in tutte le 1606 righe della colonna Sponsor: ID dell'organizzazione di appartenenza. È un bug dell'export Lions, non nostro.

La vista originale vista_report_ricerca faceva il JOIN su quell'ID, quindi tutte le attività risultavano collegate ad ABETONE.

8-bis.2 Soluzione strutturale (5 step)

#AzioneComando
1Aggiunta colonna nome_club_sponsorALTER TABLE attivita_report ADD COLUMN nome_club_sponsor TEXT;
2Popolata dai dati CSV9 batch UPDATE da 200 righe ciascuno (1606 totali) via API
3Vista ricreata con JOIN sul nomeJOIN LATERAL (SELECT … FROM club WHERE UPPER(TRIM(nome_club)) = UPPER(TRIM(r.nome_club_sponsor)) LIMIT 1)
4Script convertitore aggiornatoAggiunto mapping 'Sponsor: Nome account': 'nome_club_sponsor'
5Verifica1606 attività, 80 club distinti, 0 senza zona ✓

8-bis.3 Perché LATERAL?

La tabella club ha un duplicato letterale per ABETONE MONTAGNA PISTOIESE (2 record stessi dati, ID diversi — probabilmente uniti da officer.csv e report.csv). Senza LATERAL ... LIMIT 1, il JOIN avrebbe duplicato le righe (1606 → 1607).

-- View finale
CREATE VIEW vista_report_ricerca AS
SELECT r.*, COALESCE(c.nome_club, r.nome_club_sponsor) AS sponsor_nome_account,
       c.zona AS sponsor_zona, c.circoscrizione AS sponsor_circoscrizione
FROM attivita_report r
LEFT JOIN LATERAL (
  SELECT nome_club, zona, circoscrizione FROM club
  WHERE UPPER(TRIM(nome_club)) = UPPER(TRIM(r.nome_club_sponsor))
  LIMIT 1
) c ON TRUE;

8-ter.3 Robustezza per import futuri

Lo script convertitore_csv.py gestisce automaticamente:

  • Bug ID-club di Lions (match per nome con la nuova colonna)
  • Header con ? letterali (apostrofi smart non rappresentabili in latin1, sostituiti dall'export Lions)
  • Boolean 0/1 → true/false/NULL
  • Date in formato europeo DD/MM/YYYY → ISO
  • Numeri europei (1.234,56 → 1234.56)
  • Matricole duplicate (drop_duplicates)
  • Encoding latin1/cp1252 (lettura corretta di à, è, ecc.)

Procedura raccomandata per ogni nuovo export Lions:

cd Dati_Lions
# 1. Sostituisci i 3 file CSV con quelli nuovi
