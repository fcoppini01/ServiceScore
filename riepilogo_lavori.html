2. Bug Fix Critici

2.1 Auth callback – sessione non persistita dopo conferma email

Il file src/app/auth/callback/route.ts scriveva i cookie di sessione sulla request invece che sulla response, causando la perdita della sessione subito dopo il login via link email.

// PRIMA (sbagliato)
request.cookies.set(name, value)

// DOPO (corretto)
response.cookies.set(name, value, options)  // ← options include SameSite, HttpOnly, ecc.

2.2 Matricola socio non collegata all'account

La matricola inserita in fase di registrazione non veniva mai trasmessa al callback di conferma email. Fix: aggiunta come parametro nella URL di redirect.

const callbackUrl = new URL(`${location.origin}/auth/callback`)
if (matricola) callbackUrl.searchParams.set('matricola', matricola)

2.3 Font CSS – riferimento circolare

In globals.css le variabili CSS dei font puntavano a se stesse (var(--font-sans) si auto-referenziava). Fix: agganciate alle variabili generate da Next.js.

/* PRIMA */
--font-sans: var(--font-sans);   /* circolare! */
--font-mono: var(--font-geist-mono);  /* inesistente */

/* DOPO */
--font-sans: var(--font-ibm-sans);
--font-mono: var(--font-ibm-mono);

2.4 Messaggio di errore login mostrato in verde

La logica per scegliere il colore del messaggio controllava message.includes('Controlla'), ma quella stringa appariva anche nei messaggi di errore. Fix: controllo invertito su keyword di successo.

2.5 Hydration mismatch – particelle animate

La homepage usava Math.random() per posizionare le particelle decorative, causando un mismatch tra server-side rendering e client-side. Fix: array deterministico pre-calcolato.

2.6 TypeScript error – Recharts Tooltip formatter

Il tipo Formatter<ValueType, NameType> di Recharts non accettava la type annotation esplicita. Fix: rimossa la annotazione lasciando inferire a TypeScript.

// PRIMA – errore di build
formatter={(value: number) => [value, 'Soci']}

// DOPO – corretto
formatter={(value) => [value, 'Soci']}

4. Responsive e Paginazione Tabelle

Soluzione alla "tabella gigante": card view su mobile + paginazione server-side da 20 record + filtri collassabili.

BreakpointComportamento
< md (mobile)Card compatte per ogni record: info essenziali, badge zona/stato, dati chiave
≥ md (desktop)Tabella completa con tutte le colonne

Paginazione server-side

// Sostituisce il vecchio .limit(100)
const { data, count } = await query
  .select('*', { count: 'exact' })
  .range(page * PAGE_SIZE, (page + 1) * PAGE_SIZE - 1)  // PAGE_SIZE = 20
  • Navigazione Prev/Next con contatore "X–Y di N totali"
  • Reset automatico a pagina 0 ad ogni cambio filtro (updateFilters())
  • Debounce 300ms sulla ricerca testuale

Filtri collassabili su mobile

  • Bottone "Filtri" visibile solo su < sm, nasconde/mostra il pannello
  • Su ≥ sm i filtri sono sempre visibili
  • Pattern: className={`${filtersOpen ? 'block' : 'hidden'} sm:block`}

Implementato su: src/app/soci/page.tsx, src/app/attivita/page.tsx, src/app/officer/page.tsx

5. Autenticazione e Sicurezza

5.1 Middleware (middleware.ts)

Le restrizioni di accesso alle route sono state temporaneamente rimosse per permettere la navigazione senza account. Il middleware aggiorna solo la sessione:

// Solo refresh sessione, nessuna protezione route
await supabase.auth.getUser();
return response;

5.2 Row Level Security (RLS)

TabellaPolicy attivaRegola
clubClub visibili a utenti autenticatiSELECT per authenticated
sociSoci visibili a utenti autenticatiSELECT per authenticated – USING (true)
officer_clubOfficer visibili a utenti autenticatiSELECT per authenticated
attivita_reportReport visibili / Inserimento e Aggiornamento officerSELECT tutti auth; INSERT/UPDATE solo officer attivi

Nota: la policy originale "Soci possono vedere i propri dati" (che mostrava solo il proprio record) è stata sostituita con una policy permissiva per tutti gli autenticati, necessaria per il funzionamento della dashboard amministrativa.

5.3 Funzione di collegamento utente–socio

SQL definito in supabase_link_user_to_socio.sql: funzione link_user_to_socio(matricola) che aggiorna il campo soci.user_id dopo la conferma email.

7. Pipeline Dati CSV → Supabase

Il flusso si aggiorna ogni volta che Lions fornisce nuovi file export.

File sorgente (dalla piattaforma Lions)

  • officer.csv – incarichi officer (Latin1, separatore ;)
  • soci.csv – anagrafica soci (Latin1, separatore ;)
  • report.csv – attività di servizio (Latin1, separatore ;)

Script di conversione (Dati_Lions/convertitore_csv.py)

Produce 4 CSV normalizzati pronti per l'import in Supabase:

OutputTabella targetSorgente
1_club_supabase.csvclubofficer.csv + report.csv (union + dedup)
2_soci_supabase.csvsocisoci.csv
3_officer_supabase.csvofficer_clubofficer.csv
4_report_supabase.csvattivita_reportreport.csv

Procedura di reimport

-- 1. Svuota tutto (CASCADE elimina dipendenze in ordine)
TRUNCATE TABLE club CASCADE;

-- 2. Importa in ordine (rispetta le FK)
-- Table Editor → Import: 1_club → 2_soci → 3_officer → 4_report
